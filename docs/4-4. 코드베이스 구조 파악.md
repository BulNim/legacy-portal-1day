# 4-4. 코드베이스 구조 파악

> 이 프로젝트의 구조와 각 서비스의 책임 요약

## 프로젝트 성격

KT DS 「클린코드 & 리팩토링 실전」 과정의 실습용 코드로, 스멜이 의도적으로 심어진 레거시다. Java 파일은 총 17개, `src/test`는 존재하지 않아 특성화 테스트가 아직 없는 순수 리팩토링 이전 상태다. DB 마이그레이션 파일도 없이 `spring.jpa.hibernate.ddl-auto=create-drop`으로 Hibernate가 Entity 애노테이션 기반으로 스키마를 매 기동 시 자동 생성한다.

## 전체 구조

```
com.ktds.portal - Spring Boot 사내 업무 포털 (결재/공지/일정)
│
├── PortalApplication        진입점
├── DataSourceConfig         DB 자동 선택 (MariaDB 연결되면 사용, 안 되면 H2 인메모리 폴백)
├── DataSeeder               기동 시 사용자 4명 시드 (CommandLineRunner, 자동 실행)
│
├── approval/   ★ 주인공     결재: 생성 + 상신/승인/반려/취소 (God Class)
├── notice/                  공지: 등록 + 게시 + 긴급 메일 (REST API 미노출)
├── schedule/                일정: 등록(중복검사) + 확정 (REST API 미노출)
├── user/                    사용자: 조회만 (Service 계층 없이 Controller가 Repository 직접 호출)
└── common/                  SmtpMailSender, FileAuditLogger (콘솔 출력 스텁, 인터페이스 없음)
```

## 각 도메인의 클래스 구성

| 패키지 | Entity | Repository | Service | Controller | DTO |
|--------|--------|-------------|---------|------------|-----|
| approval | `Approval` | `ApprovalRepository` | `ApprovalService` | `ApprovalController` | 없음 (Map으로 요청 수신) |
| notice | `Notice` | `NoticeRepository` | `NoticeService` | 없음 | 없음 |
| schedule | `Schedule` | `ScheduleRepository` | `ScheduleService` | 없음 | 없음 |
| user | `User` | `UserRepository` | 없음 | `UserController` | 없음 |
| common | - | - | `FileAuditLogger`, `SmtpMailSender` | - | - |

Enum, `@Converter`/`AttributeConverter`, Config(DataSourceConfig 제외), Exception 클래스는 전무하다.

## 각 서비스의 책임

| 서비스 | 하는 일 | 상태값(int) | REST 노출 |
|--------|---------|------------|-----------|
| **ApprovalService** ★ | 결재 생성(`create`) + `processApproval`(action 1상신/2승인/3반려/9취소) + 메일 + 감사로그 + 권한판정(`role>=2`) + 라벨(`statusLabel`)/등급(`amountGrade`) 계산 + 목록 조회(`myDrafts`/`myInbox`) | 0임시저장 1상신 2승인 3반려 9취소 | O `/api/approvals` |
| **NoticeService** | 공지 생성(`create`) + 게시(`publish`, role>=2 & status==0일 때만) + 긴급(category=2)이면 전직원 메일 + 라벨(`statusLabel`) | 0임시 1게시 9내림 | X 없음 |
| **ScheduleService** | 일정 생성(`create`, **시간 겹침 검사** 포함) + 확정(`confirm`, 본인 소유 & status==0일 때만) | 0예정 1확정 9취소 | X 없음 |
| **UserController** | 사용자 목록 조회만 (`GET /api/users` -> `userRepo.findAll()`) | role: 1사원 2팀장 3임원 | O `/api/users` |
| **FileAuditLogger** | 감사로그 기록 - 실제로는 콘솔에 `[AUDIT] ...` 출력만 함 (파일 append 아님) | - | - |
| **SmtpMailSender** | 메일 발송 - 실제로는 콘솔에 TO/SUBJECT/body 출력만 함 (실제 SMTP 없음) | - | - |

## 도메인 간 의존 관계

- **approval -> user**: 결재자/기안자 조회, `role >= 2` 권한 판정에 사용
- **notice -> user**: 작성자/게시 권한 확인, 전직원 메일 발송 대상(`findAll()`) 조회에 사용
- **schedule -> user**: 확정 처리 시 사용자 존재 확인용 (role 기반 로직은 사용하지 않음, 본인 확인만)
- **approval / notice / schedule -> common**: 세 서비스 모두 `FileAuditLogger` 사용. `approval`/`notice`만 `SmtpMailSender`도 사용 (`schedule`은 메일 발송 없음)
- **approval ↔ notice ↔ schedule**: 서로 직접 참조하지 않음. user/common을 통해서만 간접 연결
- 세 서비스 모두 `mail`/`audit` 협력 객체를 생성자 주입이 아니라 필드에서 **직접 `new`**로 생성 (강결합, 테스트 시 mock 교체 불가)

## 핵심 관찰 3가지 (실습 관점)

**1. ApprovalService가 왜 "주인공"인가**

혼자서 6가지 책임을 다 짊어진 God Class다: 검증 · 영속화 · 메일발송 · 감사로그 · 권한판정(`role>=2`) · 화면표시(라벨/등급). `processApproval` 하나가 4개 액션을 if로 분기하는 100줄짜리 Long Method이며, 파라미터도 8개(`create`)에 달한다.

**2. 세 서비스에 같은 코드가 복붙됨 (중복 = 공통 모듈 추출 재료)**

```
감사로그    -> Approval / Notice / Schedule 세 곳 모두 동일한 포맷 복붙
메일 본문   -> Approval / Notice 두 곳에 유사 패턴
new FileAuditLogger()  -> 세 서비스 모두 직접 생성 (강결합)
권한 판정 role>=2      -> Approval / Notice 복붙
statusLabel(int -> 한글) -> Approval / Notice 구조가 판박이
```

이게 "중복 제거 -> `common`으로 추출" 실습의 핵심 재료다.

**3. 모든 엔티티가 빈약한 도메인 + 매직넘버**

`Approval`/`Notice`/`Schedule`/`User` 전부 데이터만 있고 행위가 없는 Anemic Domain Model이며, status·type·priority·role·category가 전부 원시 `int`다. 의미(0=임시저장 등)가 **주석에만** 있고 코드엔 숫자만 흩어져 있다. -> enum 치환 대상 (단, DB 저장값은 기존 정수 그대로 유지해야 하는 불변 규칙 적용 대상).

## 참고: 서비스별 고유 스멜

- **ApprovalService**: God Class, Long Method(`processApproval`), 파라미터 8개(`create`), 실패 조건(권한 없음 등)을 예외 없이 조용히 무시(no-op)
- **NoticeService**: 긴급 공지 메일 루프가 `publish` 메서드 안에 그대로 박혀 있음
- **ScheduleService**: 시간 겹침 검사를 의미 불명 변수명(`flag1`)의 for 루프로 처리(조건식 추출 대상), 겹치면 예외 대신 `null` 반환(호출자가 원인을 알 수 없음)
- **UserController**: Service 계층 없이 Controller가 Repository를 직접 호출 (계층 분리 누락)

## 보존 대상 vs 개선 대상 (CLAUDE.md 원칙 재확인)

- **보존 대상** (리팩토링 단계에선 그대로 유지): 권한 없는 승인/게시/확정을 조용히 무시하는 동작, 일정 겹침 시 `null` 반환하는 동작, DB에 저장되는 정수 상태값(0/1/2/3/9), 기존 REST 엔드포인트와 응답 구조
- **개선 대상** (내부 구조만 개선): God Class 분해, 매직넘버 -> enum(+ Converter), `new` 직접 생성 -> 생성자 주입, Map 요청/Entity 응답 -> DTO(record), 3개 서비스 간 중복 코드(감사로그/메일/권한판정/라벨) -> 공통 모듈 추출

---

한눈에 정리하면: **approval이 리팩토링 1순위(구조 문제), 세 서비스 공통의 메일/로그/권한판정 중복이 2순위(추출 대상), 전 엔티티의 int 매직넘버가 3순위(enum화)** 다.
