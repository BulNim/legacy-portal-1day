# 4-8. 매직넘버 탐지

> 코드에 흩어진 매직넘버를 찾아 의미를 추정하고 정리

측정 대상: `approval/`, `notice/`, `schedule/`, `user/` 전체.
매직넘버 = 이름 없이 코드에 박힌 숫자 리터럴. 의미가 **주석에만** 있고 코드엔 숫자만 남아 있는 것이 문제.

---

## 1. 결재 상태 (Approval.status) -> enum `ApprovalStatus` 후보

| 숫자 | 추정 의미 | 등장 위치(예) |
|:---:|-----------|---------------|
| `0` | 임시저장 (DRAFT) | ApprovalService.create `setStatus(0)`, processApproval `s==0` |
| `1` | 상신 (SUBMITTED) | processApproval `setStatus(1)`, `s==1` |
| `2` | 승인 (APPROVED) | processApproval `setStatus(2)` |
| `3` | 반려 (REJECTED) | processApproval `setStatus(3)` |
| `9` | 취소 (CANCELED) | processApproval `setStatus(9)`, statusLabel |

- `statusLabel()`에서 0/1/2/3/9를 "임시저장/상신/승인/반려/취소"로 번역 -> 의미 확정.
- `9`만 뛰어서 배정된 이유는 코드에 없음(4~8은 결번).

---

## 2. 결재 처리 액션 (process action) -> enum `ApprovalAction` 후보

| 숫자 | 추정 의미 | 등장 위치 |
|:---:|-----------|-----------|
| `1` | 상신 | processApproval `proc==1`, Controller 주석 |
| `2` | 승인 | `proc==2` |
| `3` | 반려 | `proc==3` |
| `9` | 취소 | `proc==9` |

- API 요청 파라미터 `action`으로 그대로 노출(컨트롤러가 매직넘버를 강요).
- status와 값이 겹치지만(1,2,3,9) **의미 축이 다름**(상태 vs 행위) -> 별도 enum 필요.

---

## 3. 결재 종류 (Approval.type) -> enum `ApprovalType` 후보

| 숫자 | 추정 의미 | 등장 위치 |
|:---:|-----------|-----------|
| `1` | 지출 | processApproval `d.getType()==1`, Controller 주석 |
| `2` | 휴가 | (주석: type 2=휴가) |
| `3` | 구매 | (주석) |
| `4` | 기타 | (주석) |

- 코드에서 실제 분기에 쓰이는 건 `type==1`(지출) 하나. 나머지는 주석으로만 존재.

---

## 4. 우선순위 (Approval.priority) -> enum `Priority` 후보

| 숫자 | 추정 의미 | 등장 위치 |
|:---:|-----------|-----------|
| `1` | 낮음 | (주석) |
| `2` | 보통 | (주석) |
| `3` | 높음 | create `urgent ? 3 : priority`, processApproval `setPriority(3)` |

- `3`(높음)만 코드에서 직접 세팅됨. 검증이 없어 `5` 같은 범위 밖 값도 통과.

---

## 5. 사용자 권한 (User.role) -> enum `Role` 후보

| 숫자 | 추정 의미 | 등장 위치 |
|:---:|-----------|-----------|
| `1` | 사원 | DataSeeder(김사원/최사원 role=1) |
| `2` | 팀장 | DataSeeder(박팀장 role=2) |
| `3` | 임원 | DataSeeder(이임원 role=3) |
| `>= 2` | **팀장 이상(승인 권한)** | ApprovalService `role>=2`(승인/반려), NoticeService `role>=2`(게시) |

- 가장 위험한 매직넘버. `role >= 2`("팀장 이상")가 3곳에 흩어져 복붙 -> `role.canApprove()` 같은 의미 메서드로 대체 대상.

---

## 6. 금액 임계값 (amount) -> 상수/도메인 규칙 후보

| 숫자 | 추정 의미 | 등장 위치 |
|:---:|-----------|-----------|
| `1000000` | 100만원 (이상이면 우선순위 상향 / 등급 A) | processApproval `amount>=1000000`, amountGrade |
| `10000000` | 1000만원 (등급 S) | amountGrade `a>=10000000` |
| `100000` | 10만원 (등급 B) | amountGrade `a>=100000` |

- `amountGrade()`가 S/A/B/C를 이 임계값들로 계산 -> 도메인 규칙(상수 또는 등급 enum)으로 분리 대상.
- 0이 세 자리 단위라 `1000000`/`10000000` 육안 구분이 어려움(가독성 문제).

---

## 7. 공지 상태/분류 (Notice) -> enum 후보

| 필드 | 숫자 | 추정 의미 | 등장 위치 |
|------|:---:|-----------|-----------|
| status | `0` | 임시(게시 전) | create `setStatus(0)`, publish `status==0` |
| status | `1` | 게시 (PUBLISHED) | publish `setStatus(1)` |
| status | `9` | 내림 | statusLabel |
| category | `1` | 일반 | (주석) |
| category | `2` | 긴급 (전직원 메일 발송) | publish `category==2` |
| category | `3` | 인사 | (주석) |

---

## 8. 일정 상태 (Schedule.status) -> enum 후보

| 숫자 | 추정 의미 | 등장 위치 |
|:---:|-----------|-----------|
| `0` | 예정 (확정 전) | create `setStatus(0)`, confirm `status==0` |
| `1` | 확정 (CONFIRMED) | confirm `setStatus(1)` |
| `9` | 취소 | create 겹침검사 `s.getStatus()!=9` |

---

## 요약

| 개념 축 | 매직넘버 | 제안 enum |
|---------|----------|-----------|
| 결재 상태 | 0/1/2/3/9 | `ApprovalStatus` |
| 결재 액션 | 1/2/3/9 | `ApprovalAction` |
| 결재 종류 | 1/2/3/4 | `ApprovalType` |
| 우선순위 | 1/2/3 | `Priority` |
| 사용자 권한 | 1/2/3, `>=2` | `Role` (+ 판정 메서드) |
| 금액 임계 | 100000 / 1000000 / 10000000 | 상수 or 등급 enum |
| 공지 상태/분류 | 0/1/9, 1/2/3 | `NoticeStatus` / `NoticeCategory` |
| 일정 상태 | 0/1/9 | `ScheduleStatus` |

**공통 문제**: 의미가 전부 **주석에만** 있고 코드엔 숫자만 있어, 주석과 코드가 어긋나면 아무도 모른다. 또한 검증이 없어 정의되지 않은 값(priority=5 등)도 통과한다.

**리팩토링 1순위**: `role >= 2`(권한)와 `ApprovalStatus`(상태)다. 권한은 3곳 복붙이라 버그 위험이 크고, 상태는 statusLabel/분기 전반에 걸쳐 가장 광범위하게 흩어져 있다. enum으로 바꾸면 컴파일러가 오타/미정의 값을 잡아준다.
