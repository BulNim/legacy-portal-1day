# 4-8. 매직넘버 탐지

> 코드에 흩어진 매직넘버를 찾아 의미를 추정하고 정리

측정 대상: `approval/`, `notice/`, `schedule/`, `user/`, `DataSeeder` 전체.
매직넘버 = 이름 없이 코드에 박힌 숫자 리터럴. 의미가 **주석에만** 있고 코드엔 숫자만 남아 있는 것이 문제.

---

## 1. 결재 상태 (Approval.status) -> enum `ApprovalStatus` 후보

| 숫자 | 추정 의미 | 등장 위치(예) |
|:---:|-----------|---------------|
| `0` | 임시저장 (DRAFT) | ApprovalService.create `setStatus(0)`, processApproval `s==0` |
| `1` | 상신 (SUBMITTED) | processApproval `setStatus(1)`, `s==1` |
| `2` | 승인 (APPROVED) | processApproval `setStatus(2)` |
| `3` | 반려 (REJECTED) | processApproval `setStatus(3)` |
| `9` | 취소 (CANCELED) | processApproval `setStatus(9)`, statusLabel |

- `statusLabel()`에서 0/1/2/3/9를 "임시저장/상신/승인/반려/취소"로 번역 -> 의미 확정.
- `9`만 뛰어서 배정된 이유는 코드에 없음(4~8은 결번).

---

## 2. 결재 처리 액션 (process action) -> enum `ApprovalAction` 후보

| 숫자 | 추정 의미 | 등장 위치 |
|:---:|-----------|-----------|
| `1` | 상신 | processApproval `proc==1`, ApprovalController 주석 |
| `2` | 승인 | `proc==2` |
| `3` | 반려 | `proc==3` |
| `9` | 취소 | `proc==9` |

- API 요청 파라미터 `action`으로 그대로 노출(`ApprovalController.process`가 매직넘버를 강요, 프론트엔드가 숫자를 외워야 함).
- status와 값이 겹치지만(1,2,3,9) **의미 축이 다름**(상태 vs 행위) -> 별도 enum 필요.

---

## 3. 결재 종류 (Approval.type) -> enum `ApprovalType` 후보

| 숫자 | 추정 의미 | 등장 위치 |
|:---:|-----------|-----------|
| `1` | 지출 | processApproval `d.getType()==1`, ApprovalController 주석 |
| `2` | 휴가 | (주석) |
| `3` | 구매 | (주석) |
| `4` | 기타 | (주석) |

- 코드에서 실제 분기에 쓰이는 건 `type==1`(지출) 하나. 나머지는 주석으로만 존재.
- `ApprovalController.create`가 요청 Map에서 `(int) body.get("type")`을 그대로 꺼내 검증 없이 넘김 -> 범위 밖 값(예: `9`)도 그냥 저장됨.

---

## 4. 우선순위 (Approval.priority) -> enum `Priority` 후보

| 숫자 | 추정 의미 | 등장 위치 |
|:---:|-----------|-----------|
| `1` | 낮음 | (주석) |
| `2` | 보통 | (주석) |
| `3` | 높음 | create `urgent ? 3 : priority`, processApproval `setPriority(3)` |

- `3`(높음)만 코드에서 직접 세팅됨. 검증이 없어 `5` 같은 범위 밖 값도 통과.
- `urgent ? 3 : priority` - `urgent`가 true면 요청받은 priority 값을 무시하고 `3`으로 덮어씀. 이 규칙이 숫자 `3` 하나로만 표현되어 있어 "왜 3인지"는 옆의 `d.setPriority(3)`(processApproval, 100만원 이상 자동 상향)과 대조해야만 알 수 있다.

---

## 5. 사용자 권한 (User.role) -> enum `Role` 후보

| 숫자 | 추정 의미 | 등장 위치 |
|:---:|-----------|-----------|
| `1` | 사원 | DataSeeder(김사원/최사원 role=1) |
| `2` | 팀장 | DataSeeder(박팀장 role=2) |
| `3` | 임원 | DataSeeder(이임원 role=3) |
| `>= 2` | **팀장 이상(승인 권한)** | ApprovalService `role>=2`(승인/반려), NoticeService `role>=2`(게시) |

- 가장 위험한 매직넘버. `role >= 2`("팀장 이상")가 ApprovalService(승인)·ApprovalService(반려)·NoticeService(게시) 3곳에 흩어져 복붙 -> `role.canApprove()` 같은 의미 메서드로 대체 대상.
- `DataSeeder`가 초기 사용자 4명에 role 값을 정수 리터럴(1,2,3,1)로 직접 박아 넣음 - enum이 생기면 이 시딩 코드도 함께 손봐야 하는 지점.

---

## 6. 금액 임계값 (Approval.amount) -> 상수/도메인 규칙 후보

| 숫자 | 추정 의미 | 등장 위치 |
|:---:|-----------|-----------|
| `1000000` | 100만원 (이상이면 우선순위 자동 상향 `3`(높음) / 등급 A) | processApproval `amount>=1000000`, amountGrade |
| `10000000` | 1000만원 (등급 S) | amountGrade `a>=10000000` |
| `100000` | 10만원 (등급 B) | amountGrade `a>=100000` |

- `amountGrade()`가 S/A/B/C를 이 임계값들로 계산 -> 도메인 규칙(상수 또는 등급 enum)으로 분리 대상.
- `processApproval`의 `amount>=1000000` 우선순위 자동 상향 기준과 `amountGrade`의 A 등급 기준이 같은 값(`1000000`)인데도 서로 다른 곳에 따로 적혀 있어, 하나만 고치면 다른 하나는 놓치기 쉽다(값의 의미적 연결이 코드에 드러나지 않음).
- 0이 세 자리 단위라 `1000000`/`10000000` 육안 구분이 어려움(가독성 문제).

---

## 7. 공지 상태/분류 (Notice) -> enum 후보

| 필드 | 숫자 | 추정 의미 | 등장 위치 |
|------|:---:|-----------|-----------|
| status | `0` | 임시(게시 전) | create `setStatus(0)`, publish `status==0` |
| status | `1` | 게시 (PUBLISHED) | publish `setStatus(1)` |
| status | `9` | 내림 | statusLabel |
| category | `1` | 일반 | (주석) |
| category | `2` | 긴급 (전직원 메일 발송) | publish `category==2` |
| category | `3` | 인사 | (주석) |

- `category==2`(긴급)일 때만 전직원 메일 발송 분기를 탐 - 나머지 category(1, 3)는 코드에서 실제로 갈리지 않고 주석으로만 존재.
- Notice.status(0/1/9)가 Approval.status(0/1/2/3/9)와 값 일부가 겹치지만 **서로 다른 도메인의 별개 의미** -> 하나의 공용 상수로 묶으면 안 되고 도메인별 enum이 맞다.

---

## 8. 일정 상태 (Schedule.status) -> enum `ScheduleStatus` 후보

| 숫자 | 추정 의미 | 등장 위치 |
|:---:|-----------|-----------|
| `0` | 예정 | create `setStatus(0)`, confirm `status==0` |
| `1` | 확정 | confirm `setStatus(1)` |
| `9` | 취소 | ScheduleService.create 겹침 검사 `s.getStatus() != 9` |

- 세 도메인(Approval/Notice/Schedule) 모두 `9 = 취소/내림`을 관례처럼 재사용하지만, 이 관례 자체가 코드 어디에도 문서화되어 있지 않다(각 클래스 상단 주석에만 개별적으로 적혀 있음).
- 겹침 검사에서 `status != 9`(취소된 일정 제외)만 boolean 매직넘버 없이 조건식에 바로 박혀 있어, "취소된 일정은 겹침 계산에서 빠진다"는 규칙이 숫자 하나로 숨어 있다.

---

## 9. 요약

| 매직넘버 그룹 | enum 후보 | 위험도 |
|---|---|:---:|
| Approval.status (0/1/2/3/9) | `ApprovalStatus` | 상 |
| process action (1/2/3/9) | `ApprovalAction` | 상 |
| Approval.type (1~4) | `ApprovalType` | 중 |
| Approval.priority (1~3) | `Priority` | 중 |
| User.role (1~3, `>=2` 판정) | `Role` | **최상** |
| amount 임계값 (100000/1000000/10000000) | 등급 상수/`AmountGrade` | 상 |
| Notice.status (0/1/9) | `NoticeStatus` | 중 |
| Notice.category (1~3) | `NoticeCategory` | 하(분기 1곳뿐) |
| Schedule.status (0/1/9) | `ScheduleStatus` | 중 |

- **가장 시급한 건 `role >= 2`다.** 같은 판정(팀장 이상)이 서로 다른 서비스 3곳에 복붙되어 있어, 권한 기준이 바뀌면 3곳을 다 찾아 고쳐야 한다.
- 그다음은 **`Approval.status`/`process action`** - 값 축이 겹치면서(1,2,3,9) 의미가 다른 두 매직넘버가 나란히 쓰여 헷갈리기 쉽고, `action`은 API로도 그대로 노출된다.
- `CLAUDE.md`의 불변 규칙에 따라, enum으로 바꾸더라도 **DB 저장값(0/1/2/3/9 등 정수)과 API 요청/응답 형식은 그대로 유지**해야 한다 - `@Convert` + `AttributeConverter`로 code 매핑하고, `@Enumerated(STRING/ORDINAL)`은 쓰지 않는다.
