# 4-9. 강결합 탐지

> 생성자 주입 없이 직접 `new` 로 생성하는 의존성(협력 객체) 탐지

강결합 = 서비스가 협력 객체(메일 발송기, 감사 로거 등)를 **직접 `new`** 로 만드는 것.
바깥에서 교체할 수 없어 테스트에서 가짜(mock)로 갈아끼우지 못한다.

> 참고: `new Approval()`, `new Notice()`, `new Schedule()` 처럼 **데이터/엔티티 객체를 만드는 new는 강결합이 아니다.** 문제는 "협력 객체(서비스성 의존성)"를 직접 new 하는 경우다.

---

## 1. 직접 `new` 로 생성하는 협력 객체 (전수)

| # | 파일 | 필드 | 코드 | 대상 클래스 |
|:---:|------|------|------|-------------|
| 1 | ApprovalService | `mail` | `private final SmtpMailSender mail = new SmtpMailSender();` | SmtpMailSender |
| 2 | ApprovalService | `audit` | `private final FileAuditLogger audit = new FileAuditLogger();` | FileAuditLogger |
| 3 | NoticeService | `mail` | `private final SmtpMailSender mail = new SmtpMailSender();` | SmtpMailSender |
| 4 | NoticeService | `audit` | `private final FileAuditLogger audit = new FileAuditLogger();` | FileAuditLogger |
| 5 | ScheduleService | `audit` | `private final FileAuditLogger audit = new FileAuditLogger();` | FileAuditLogger |

**총 5건.** 두 종류의 협력 객체가 세 서비스에 흩어져 직접 생성되고 있다.

### 대상별 집계

| 협력 객체 | 직접 new 횟수 | 위치 |
|-----------|:---:|------|
| `FileAuditLogger` | **3** | ApprovalService, NoticeService, ScheduleService |
| `SmtpMailSender` | **2** | ApprovalService, NoticeService |

---

## 2. 정상(주입된) 의존성과 대비

같은 서비스 안에서도 **레포지토리는 생성자 주입**을 받는데, mail/audit만 직접 new 한다. 일관성이 없다.

```java
public class ApprovalService {
    private final ApprovalRepository repo;      // O 생성자 주입
    private final UserRepository userRepo;       // O 생성자 주입

    private final SmtpMailSender mail  = new SmtpMailSender();    // X 직접 new
    private final FileAuditLogger audit = new FileAuditLogger();  // X 직접 new

    public ApprovalService(ApprovalRepository repo, UserRepository userRepo) {
        this.repo = repo;
        this.userRepo = userRepo;
        // mail, audit 은 생성자에 없음 -> 바깥에서 통제 불가
    }
}
```

- `repo`/`userRepo` : 생성자 파라미터 -> 테스트에서 mock 주입 가능.
- `mail`/`audit` : 클래스 안에서 박제 -> 교체 불가.

---

## 3. 왜 문제인가

1. **테스트 불가** : `ApprovalService`를 테스트하면 `mail.send(...)`가 실제로 실행된다(이 실습은 콘솔 출력이지만, 실무라면 진짜 SMTP 발송). 가짜로 바꿔 "호출 여부만" 검증할 수 없다.
2. **구현 교체 불가** : 로그 대상을 파일 -> DB로 바꾸려면 세 서비스의 `new FileAuditLogger()`를 전부 찾아 고쳐야 한다.
3. **구체 클래스 의존** : 인터페이스가 아니라 구체 클래스(`SmtpMailSender`, `FileAuditLogger`)에 직접 묶여 있다(DIP 위반).

---

## 4. 리팩토링 방향 (DIP 적용)

**1단계 - 인터페이스 추출**
```java
public interface MailSender  { void send(String to, String subject, String body); }
public interface AuditLogger { void write(String line); }

// 구현체
public class SmtpMailSender  implements MailSender  { ... }
public class FileAuditLogger implements AuditLogger { ... }
```

**2단계 - 생성자 주입으로 전환** (직접 new 제거)
```java
public class ApprovalService {
    private final MailSender mail;        // 구체 클래스 -> 인터페이스
    private final AuditLogger audit;

    public ApprovalService(ApprovalRepository repo, UserRepository userRepo,
                           MailSender mail, AuditLogger audit) {   // 전부 주입
        this.repo = repo;
        this.userRepo = userRepo;
        this.mail = mail;
        this.audit = audit;
    }
}
```

**결과**: `new SmtpMailSender()` / `new FileAuditLogger()` 5건이 전부 사라지고, 테스트에서 `new ApprovalService(mockRepo, mockUserRepo, fakeMail, fakeAudit)`로 가짜 주입이 가능해진다.

---

## 요약

| 항목 | 값 |
|------|-----|
| 직접 new 협력 객체 총 건수 | **5** |
| 관련 서비스 | ApprovalService, NoticeService, ScheduleService (전부) |
| 대상 클래스 | `FileAuditLogger`(3), `SmtpMailSender`(2) |
| 근본 원인 | 인터페이스 부재 + 생성자 주입 누락(DIP 위반) |
| 처방 | 인터페이스 추출 -> 생성자 주입으로 전환 |

이 강결합은 [중복 코드 탐지]의 "협력 객체 직접 생성" 항목과 같은 뿌리다. 인터페이스화 + 주입으로 바꾸면 강결합과 중복 new가 **동시에** 해소된다.
