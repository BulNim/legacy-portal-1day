# 4-5. 중복 코드 탐지

> 세 서비스(ApprovalService, NoticeService, ScheduleService)에서 중복된 코드 분석

대상 파일:
- `approval/ApprovalService.java`
- `notice/NoticeService.java`
- `schedule/ScheduleService.java`

---

## 중복 1. 감사 로그 기록 (가장 심각 - 3곳 전부)

세 서비스가 **똑같은 타임스탬프 포맷 + audit.write 조립**을 각자 복붙하고 있다.

| 위치 | 코드 |
|------|------|
| ApprovalService.create() | `String now = LocalDateTime.now().format(DateTimeFormatter.ofPattern("yyyy-MM-dd HH:mm:ss"));` |
| ApprovalService.writeAudit() | 동일 |
| NoticeService.create() | 동일 |
| NoticeService.publish() | 동일 |
| ScheduleService.create() | 동일 |
| ScheduleService.confirm() | 동일 |

공통 패턴:
```java
String now = LocalDateTime.now().format(DateTimeFormatter.ofPattern("yyyy-MM-dd HH:mm:ss"));
audit.write("[" + now + "] " + <액션> + " id=" + <id> + " by=" + <userId>);
```

- **같은 포맷 문자열 `"yyyy-MM-dd HH:mm:ss"` 이 6번 등장**한다.
- `[시각] 액션 id=.. by=..` 형태의 로그 라인 조립 규칙도 동일하다.
- ApprovalService는 `writeAudit()`로 일부만 추출했지만 `create()` 안에는 여전히 복붙이 남아 **불완전한 중복 제거** 상태다.

**-> 추출 대상**: `common` 패키지에 `AuditService`(또는 로그 라인 포맷터)를 만들어 한 곳으로 모은다.

## 중복 2. 협력 객체 직접 생성 (강결합 - 3곳 전부)

세 서비스 모두 `FileAuditLogger`를 생성자 주입 없이 직접 `new` 한다.

```java
// ApprovalService
private final SmtpMailSender mail = new SmtpMailSender();
private final FileAuditLogger audit = new FileAuditLogger();

// NoticeService
private final SmtpMailSender mail = new SmtpMailSender();
private final FileAuditLogger audit = new FileAuditLogger();

// ScheduleService
private final FileAuditLogger audit = new FileAuditLogger();
```

- `new FileAuditLogger()` = **3곳 중복**
- `new SmtpMailSender()` = ApprovalService, NoticeService **2곳 중복**
- 구체 클래스에 강결합되어 테스트에서 교체 불가.

**-> 추출 대상**: 인터페이스(`AuditLogger`, `MailSender`)로 추출 + 생성자 주입(DIP)으로 전환하면 중복 `new` 자체가 사라진다.

## 중복 3. 메일 본문 조립 (2곳 - Approval, Notice)

"안녕하세요 OOO님," 으로 시작하는 메일 본문 생성 패턴이 반복된다.

| 위치 | 본문 패턴 |
|------|----------|
| ApprovalService (상신) | `"안녕하세요 " + approver.getName() + "님,\n결재 요청이 도착했습니다.\n제목: " + ...` |
| ApprovalService (승인) | `"안녕하세요 " + drafter.getName() + "님,\n결재가 승인되었습니다.\n제목: " + ...` |
| ApprovalService (반려) | `"안녕하세요 " + drafter.getName() + "님,\n결재가 반려되었습니다.\n제목: " + ...` |
| NoticeService (긴급공지) | `"안녕하세요 " + member.getName() + "님,\n긴급 공지가 게시되었습니다.\n제목: " + ...` |

공통 뼈대:
```java
String body = "안녕하세요 " + <수신자>.getName() + "님,\n"
            + <상황 문구> + "\n제목: " + <제목>;
mail.send(<수신자>.getEmail(), "[" + <말머리> + "] " + <제목>, body);
```

- ApprovalService 안에서만 이 패턴이 **3번 반복**(상신/승인/반려)되고, NoticeService에도 유사하게 등장한다.
- 제목도 `"[결재요청] " / "[결재승인] " / "[결재반려] " / "[긴급공지] "` 처럼 `[말머리] + 제목` 규칙이 동일.

**-> 추출 대상**: `MailMessageFactory` 같은 본문/제목 생성기로 모은다.

## 중복 4. 조회 후 null 체크 가드 (3곳 전부)

"엔티티 조회 -> null 이면 조용히 return" 패턴이 반복된다.

```java
// ApprovalService.processApproval
Approval d = repo.findById(id).orElse(null);
if (d == null) return;
User u = userRepo.findById(userId).orElse(null);
if (u == null) return;

// NoticeService.publish
Notice n = repo.findById(id).orElse(null);
if (n == null) return;
User u = userRepo.findById(userId).orElse(null);
if (u == null) return;

// ScheduleService.confirm
Schedule sc = repo.findById(id).orElse(null);
if (sc == null) return;
User u = userRepo.findById(userId).orElse(null);
if (u == null) return;
```

- `findById(...).orElse(null)` + `if (== null) return;` 구조가 **3개 메서드에 동일**.
- 특히 `userRepo.findById(userId).orElse(null); if (u == null) return;` 은 글자까지 똑같다.
- 부가 문제: 실패를 조용히 삼켜(예외 없음) 호출자가 원인을 모른다.

**-> 개선 방향**: `orElseThrow(...)`로 통일하면 null 가드 중복이 제거되고 실패 원인도 드러난다.

## 중복 5. 권한 판정 매직넘버 (2곳 - Approval, Notice)

"팀장 이상" 권한 판정을 `role >= 2` 매직넘버로 복붙한다.

```java
// ApprovalService (승인/반려 분기 각각)
if (u.getRole() >= 2) { ... }   // 2번 등장

// NoticeService.publish
if (u.getRole() >= 2) { ... }
```

- `role >= 2` 라는 판정이 여러 곳에 흩어져 있고, "2 = 팀장"의 의미는 주석에만 있다.

**-> 개선 방향**: `Role` enum + `role.canApprove()` 같은 의미 있는 판정 메서드로 일원화.

## 요약

| # | 중복 항목 | 등장 위치 | 심각도 | 추출/개선 방향 |
|---|-----------|-----------|--------|----------------|
| 1 | 감사 로그 조립(포맷 문자열 포함) | 3서비스 6곳 | 높음 | `common`에 AuditService 추출 |
| 2 | 협력 객체 직접 `new` | 3서비스 | 높음 | 인터페이스화 + 생성자 주입(DIP) |
| 3 | 메일 본문/제목 조립 | Approval 3곳 + Notice | 중간 | MailMessageFactory 추출 |
| 4 | 조회 후 null 체크 가드 | 3서비스 | 중간 | `orElseThrow`로 통일 |
| 5 | 권한 판정 `role >= 2` | Approval 2곳 + Notice | 중간 | Role enum 판정 메서드 |

**우선순위**: 1번(감사 로그)과 2번(강결합)이 세 서비스 전부에 걸쳐 있어 **공통 모듈 추출 실습의 1순위**다. 이 둘을 `common`으로 뽑아내면 나머지 3~5번 정리도 자연스럽게 이어진다.
