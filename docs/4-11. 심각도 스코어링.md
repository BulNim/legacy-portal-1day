# 4-11. 심각도 스코어링

> `docs/smells.md`(= `4-10. 도출 스멜 - 심각도 표.md`)에서 정성적 등급(최상/상/중/하)으로 나눴던 20개 스멜을, **영향도 x 수정난이도** 두 축의 정량 점수로 재채점한다.

## 채점 기준

### 1) 영향도 (Impact, 1~5점)

스멜을 방치했을 때 실무에서 발생하는 피해의 범위와 크기. 아래 관점을 종합한다.

- **범위** - 몇 개 서비스/파일에 걸쳐 있는가
- **파급력** - 기준·값이 바뀔 때 놓치기 쉬운 곳이 몇 곳인가, 장애·보안사고로 이어질 가능성
- **구조적 위치** - 다른 스멜(중복·강결합·매직넘버)의 근본 원인이 되는가

| 점수 | 기준 |
|:---:|------|
| 5 | 여러 도메인에 걸친 근본 원인, 방치 시 장애/보안사고 직결 |
| 4 | 전 도메인 또는 핵심 로직에 광범위하게 퍼져 있음 |
| 3 | 여러 서비스에 걸친 중복/구조 문제, 기준 변경 시 누락 위험 |
| 2 | 국소적이지만 가독성·유지보수에 영향 |
| 1 | 지엽적, 영향 범위 좁음 |

### 2) 수정난이도 (Difficulty, 1~5점)

리팩토링에 필요한 변경 범위와 리스크. 아래 관점을 종합한다.

- **변경 범위** - 손대야 할 클래스/메서드 수
- **동작 보존 리스크** - 특성화 테스트 없이는 observable behavior가 깨질 위험(CLAUDE.md 불변 규칙 - DB 저장값, API 계약 등)
- **선행 의존성** - 다른 스멜(예: God Class 분해)이 먼저 끝나야 손대기 쉬운가

| 점수 | 기준 |
|:---:|------|
| 5 | 클래스 전체 재설계, 여러 협력 객체 분해, 선행 리팩토링 필수 |
| 4 | 메서드 단위 대규모 분해 또는 도메인 전반의 구조 변경 |
| 3 | 여러 파일에 걸친 enum/DTO 도입 + 동작 보존 검증 필요 |
| 2 | 기계적 추출/치환(Extract Method·Class, 생성자 주입) 위주 |
| 1 | 단일 지점, 이름 변경 수준의 국소 수정 |

### 3) 심각도 점수와 별점 환산

**심각도 점수 = 영향도 x 수정난이도** (최대 5 x 5 = 25점)

고치기 쉬운데 아직 방치된 스멜보다, **영향도가 크면서 손대기도 어려운 스멜**이 "방치될 위험이 크고 방치됐을 때 리스크도 큰" 항목이라고 보고 곱셈으로 산정한다.

| 점수 구간 | 별점 |
|:---:|:---:|
| 20~25 | ★★★★★ |
| 14~19 | ★★★★☆ |
| 8~13 | ★★★☆☆ |
| 4~7 | ★★☆☆☆ |
| 1~3 | ★☆☆☆☆ |

## 4-10 등급표와 달라지는 지점

- **3번(매직넘버 - 권한 판정 복붙)** 은 4-10에서 "최상"이었지만, 여기서는 6위(★★★☆☆)로 내려간다. 영향도(파급력)는 최상급이지만 `Role` enum 도입 자체가 기계적이라 수정난이도가 낮기 때문 - 곱셈 방식에서는 "쉽게 고칠 수 있는데 아직 방치된" 항목이 God Class처럼 "고치기도 어렵고 영향도도 큰" 항목보다 낮게 나온다.
- **8번(Primitive Obsession)** 은 4-10에서 5개 항목("상")과 동급이었지만, 여기서는 3위(★★★★☆)로 올라간다. 전 도메인에 걸친 범위(영향도 4)와 도메인별 전수 적용이 필요한 난이도(4)가 곱해지며 God Class·Long Method 다음으로 부상한다.
- **14번(Anemic Domain Model)** 도 마찬가지로 "중" 등급이었지만 5위(★★★☆☆)로 상승 - 전 엔티티 재설계급 난이도가 반영된 결과다.
- 반대로 **1~2번(God Class, Long Method)** 은 두 표에서 동일하게 최상위를 유지한다. 영향도와 난이도가 모두 최고 수준이라는 데 두 채점 방식이 일치한다.

## 결론 - 착수 순서 제안

- **선행 필수(★★★★★)**: God Class 분해, `processApproval` 액션별 분리. 이 둘을 먼저 풀어야 이후 항목(강결합 제거, 중복 제거)이 더 쉬워진다.
- **구조 전환(★★★★☆~★★★☆☆, 3~9위)**: Primitive Obsession·매직넘버 계열·Anemic Domain Model·DTO 도입은 서로 얽혀 있어 enum/Converter/DTO 작업을 한 묶음으로 진행하는 편이 효율적이다.
- **정형적 정리(★★☆☆☆, 10~16위)**: `common` 모듈 추출(MailSender/AuditLogger 인터페이스, AuditService, MailMessageFactory)로 여러 항목이 동시에 해소된다.
- **후순위(★☆☆☆☆, 17~20위)**: 여유 있을 때 정리해도 무방한 지엽적 항목.
