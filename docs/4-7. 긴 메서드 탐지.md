# 4-7. 긴 메서드 탐지

> 50줄 넘는 메서드와 중첩 깊이 분석

측정 기준:
- **길이** = 메서드 시그니처 줄부터 닫는 `}` 까지 포함(주석·빈 줄 포함).
- **중첩 깊이** = 메서드 본문을 0으로 두고, 안쪽으로 들어가는 `if`/`for` 블록의 최대 깊이.
- 대상: `ApprovalService`, `NoticeService`, `ScheduleService`.

---

## 1. 50줄을 넘는 메서드 (임계 초과)

| 메서드 | 파일 | 줄 수 | 최대 중첩 깊이 | 비고 |
|--------|------|------:|:---:|------|
| `processApproval(...)` | ApprovalService | **86** | **5** | 상신/승인/반려/취소를 한 메서드가 if로 분기 |

**임계(50줄)를 넘는 메서드는 `processApproval` 단 하나다.** 하지만 이 하나가 God Method의 표본이다.

### processApproval 중첩 구조 (깊이 5)

```
processApproval() ..................... depth 0
├─ if (d == null) return; ............. 1
├─ if (u == null) return; ............. 1
├─ if (proc == 1) 상신 ................ 1
│   └─ if (s == 0) .................... 2
│       ├─ if (type==1 && amount>=100만) 3
│       └─ if (approver != null) ...... 3
├─ else if (proc == 2) 승인 ........... 1
│   └─ if (s == 1) .................... 2
│       └─ if (approverId==userId) .... 3
│           └─ if (role >= 2) ......... 4
│               └─ if (drafter != null) 5   <- 최대 깊이
├─ else if (proc == 3) 반려 ........... 1
│   └─ if (s == 1) .................... 2
│       └─ if (approverId==userId) .... 3
│           └─ if (role >= 2) ......... 4
│               └─ if (drafter != null) 5   <- 최대 깊이
└─ else if (proc == 9) 취소 ........... 1
    └─ if (s==0 || s==1) .............. 2
        └─ if (drafterId==userId) ..... 3
```

- 4개 액션(상신/승인/반려/취소)이 하나의 메서드에 뭉쳐 있다.
- 승인/반려 경로가 **5단 중첩** (액션 -> 상태 -> 본인확인 -> 권한 -> 대상 null 체크).
- 승인 분기와 반려 분기의 안쪽 3~5단이 거의 동일하게 **복붙**되어 있다.

## 2. 임계 미만이지만 주의 대상 (Watch List)

50줄은 안 넘지만 길거나 깊어서 다음 리팩토링 후보다.

| 메서드 | 파일 | 줄 수 | 최대 중첩 깊이 | 비고 |
|--------|------|------:|:---:|------|
| `create(...)` | ScheduleService | 39 | 3 | 검증+겹침검사(for 안 if)+저장+로그 한 덩어리 |
| `publish(...)` | NoticeService | 25 | 4 | 권한->상태->긴급->전직원 for 4단 중첩 |
| `create(...)` | ApprovalService | 22 | 1 | 파라미터 8개, set 나열 + 로그 복붙 |
| `create(...)` | NoticeService | 16 | 1 | set 나열 + 로그 복붙 |
| `confirm(...)` | ScheduleService | 14 | 2 | 조회 가드 + 본인/상태 확인 |
| `statusLabel(...)` | ApprovalService | 11 | 1 | if-else 라벨 번역(중복) |

- **ScheduleService.create(39줄)** 가 사실상 2위. `flag1` for 루프의 겹침 검사가 길이의 상당 부분을 차지한다.
- **NoticeService.publish(25줄)** 는 짧지만 깊이 4단으로, 길이보다 중첩이 문제다.

## 요약

- **50줄 초과 = `processApproval`(86줄) 하나뿐**이지만, 최대 중첩 5단으로 God Method의 전형이다.
- 리팩토링 1순위는 명확히 `processApproval` -> **액션별 메서드(submit/approve/reject/cancel)로 분해**해 깊이를 낮추는 것.
- 2순위는 `ScheduleService.create`(39줄, 겹침 검사 추출)와 `NoticeService.publish`(깊이 4, 긴급 메일 루프 추출).
- 공통 처방: **깊은 중첩 -> 조기 반환(guard clause)** + **긴 조건식/블록 -> 의미 있는 메서드로 추출**.
