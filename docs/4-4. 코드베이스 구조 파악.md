# 4-4. 코드베이스 구조 파악

> 이 프로젝트의 구조와 각 서비스의 책임 요약

## 전체 구조

```
com.ktds.portal - Spring Boot 사내 업무 포털 (결재/공지/일정)
│
├── PortalApplication        진입점
├── DataSourceConfig         DB 자동 선택 (MariaDB 연결되면 사용, 안 되면 H2)
├── DataSeeder               기동 시 사용자 4명 시드 (자동 실행)
│
├── approval/   ★ 주인공     결재: 생성 + 상신/승인/반려/취소 (God Class)
├── notice/                  공지: 등록 + 게시 + 긴급 메일
├── schedule/                일정: 등록(중복검사) + 확정
├── user/                    사용자: 조회만
└── common/                  SmtpMailSender, FileAuditLogger (콘솔 출력 스텁)
```

## 각 서비스의 책임

| 서비스 | 하는 일 | 상태값(int) | REST 노출 |
|--------|---------|------------|-----------|
| **ApprovalService** ★ | 결재 생성 + `processApproval`(상신1/승인2/반려3/취소9) + 메일 + 감사로그 + 권한판정 + 라벨/등급 계산 | 0임시 1상신 2승인 3반려 9취소 | O `/api/approvals` |
| **NoticeService** | 공지 생성 + 게시(`publish`) + 긴급(category=2)이면 전직원 메일 | 0임시 1게시 9내림 | X 없음 |
| **ScheduleService** | 일정 생성(**시간 겹침 검사**) + 확정(`confirm`) | 0예정 1확정 9취소 | X 없음 |
| **UserController** | 사용자 목록 조회만 | role: 1사원 2팀장 3임원 | O `/api/users` |

## 핵심 관찰 3가지 (실습 관점)

**1. ApprovalService가 왜 "주인공"인가**

혼자서 6가지 책임을 다 짊어진 God Class다: 검증 · 영속화 · 메일발송 · 감사로그 · 권한판정(`role>=2`) · 화면표시(라벨/등급). `processApproval` 하나가 4개 액션을 if로 분기하는 100줄짜리 Long Method다.

**2. 세 서비스에 같은 코드가 복붙됨 (중복 = 공통 모듈 추출 재료)**

```
감사로그    -> Approval / Notice / Schedule 세 곳 모두 동일한 6줄 복붙
메일 본문   -> Approval / Notice 두 곳에 유사 패턴
new FileAuditLogger()  -> 세 서비스 모두 직접 생성 (강결합)
권한 판정 role>=2      -> Approval / Notice 복붙
```

이게 "중복 제거 -> `common`으로 추출" 실습의 핵심 재료다.

**3. 모든 엔티티가 빈약한 도메인 + 매직넘버**

`Approval`/`Notice`/`Schedule`/`User` 전부 데이터만 있고 행위가 없으며, status·type·role·category가 전부 `int`다. 의미(0=임시저장 등)가 **주석에만** 있고 코드엔 숫자만 흩어져 있다. -> enum 치환 대상.

## 참고: 서비스별 고유 스멜

- **ApprovalService**: God Class, Long Method, 파라미터 8개, 변수명 `d`/`s`/`proc`
- **NoticeService**: 긴급 공지 메일 루프가 `publish` 안에 박힘
- **ScheduleService**: 시간 겹침 검사를 `flag1`이라는 for 루프로 (조건식 추출 대상), 실패 시 `null` 반환

---

한눈에 정리하면: **approval이 리팩토링 1순위(구조 문제), 세 서비스 공통의 메일/로그 중복이 2순위(추출 대상), 전 엔티티의 int 매직넘버가 3순위(enum화)** 다.
