# 4-12. 리팩토링 백로그

> `docs/4-11. 심각도 스코어링.md`의 점수(영향도 x 수정난이도)를 근거로, 실제 작업 순서를 백로그(작업 항목)로 재구성한다.
> 4-11이 "스멜 20개를 개별 채점"한 표라면, 이 문서는 "묶어서 실행할 단위(티켓)"로 다시 쪼갠 작업 목록이다.

## 백로그 구성 원칙

- **점수 순서를 그대로 따르지 않는다.** 같은 리팩토링 기법(enum 도입, 인터페이스 추출 등)으로 동시에 해소되는 스멜은 하나의 티켓으로 묶는다. 예: 3위 Primitive Obsession과 4위 매직넘버(상태/행위 값 축 겹침)는 같은 "enum + `@Convert`" 작업 안에서 함께 처리된다.
- **선행 필수(★★★★★) → 구조 전환(★★★★☆~★★★☆☆) → 정형적 정리(★★☆☆☆) → 후순위(★☆☆☆☆)** 4단계는 4-11의 결론을 그대로 승계한다.
- 각 티켓은 착수 전 **CLAUDE.md 불변 규칙**(API 계약·DB 저장값·observable behavior 보존)을 반드시 점검하고, 착수 조건으로 **특성화 테스트 확보 여부**를 명시한다.
- 완료 조건은 공통으로 다음을 만족해야 한다: **관련 특성화 테스트가 수정 없이 green**, **DB 저장값(정수) 불변**, **API 엔드포인트/요청·응답 구조 불변**.

## 특성화 테스트 현황 (선행 점검)

| 도메인 | 특성화 테스트 | 상태 |
|--------|----------------|------|
| Approval | `ApprovalServiceCharacterizationTest` | 존재 - `processApproval` 상신/승인/반려/취소 4액션 + role 권한 분기 커버 |
| Notice | 없음 | **RF-00에서 작성 필요** |
| Schedule | 없음 | **RF-00에서 작성 필요** |
| User | 없음 | 조회 전용(계층 분리 누락 항목만 해당) - RF-11 착수 전 최소 테스트 필요 |

> Notice·Schedule을 건드리는 티켓(RF-02, RF-04, RF-05, RF-07 등)은 RF-00이 끝나기 전까지 "동작 보존을 검증할 안전망이 없는" 상태다. **순서를 앞당기지 말 것.**

## Phase 0 - 착수 전 준비 (선행 필수)

### RF-00. Notice/Schedule 특성화 테스트 작성
- **대상**: `NoticeService`, `ScheduleService`
- **관련 4-11 순위**: 전체 티켓의 선행 조건 (자체 순위 없음)
- **작업**: `ApprovalServiceCharacterizationTest`와 동일한 방식으로, 현재 관찰되는 동작(권한 판정 `role>=2`, 상태 전이, 조용한 무시 케이스, 겹침 검사 등)을 값으로 고정
- **완료 조건**: 이후 모든 Notice/Schedule 관련 리팩토링(RF-02, RF-04, RF-05, RF-07)의 전제 조건 충족
- **주의**: 결함(권한 없는 처리를 조용히 무시하는 등)도 "지금 이렇게 동작한다"는 사실로만 고정한다. 옳고 그름을 판단해 테스트를 작성하지 않는다.

## Phase 1 - 선행 필수 (★★★★★, 4-11 1~2위)

### RF-01. `ApprovalService` God Class 분해
- **관련 4-11 순위**: 1위 (영향도5 x 난이도5 = 25점)
- **선행 조건**: 없음 (이미 특성화 테스트 확보됨)
- **작업**: 검증·영속화·메일·감사로그·권한판정·화면표시 6개 책임을 협력 객체로 분리
- **불변 규칙**: `processApproval(id, userId, action, reason)` public 시그니처 유지, `/api/approvals/{id}/process` 엔드포인트 유지
- **후속 영향**: RF-03(enum 도입), RF-08(DIP), RF-09(중복 제거)가 이 분해 위에서 진행되면 충돌이 적음 - **가장 먼저 착수**

### RF-02. `processApproval` 액션별 메서드 분리
- **관련 4-11 순위**: 2위 (5 x 4 = 20점)
- **선행 조건**: RF-01과 함께 진행(같은 클래스, 같은 리팩토링 세션 권장)
- **작업**: 86줄·5단 중첩 메서드를 상신(submit)/승인(approve)/반려(reject)/취소(cancel)로 분리, guard clause로 중첩 완화
- **불변 규칙**: 외부에서는 여전히 `processApproval` 하나로 호출(내부에서만 액션별 위임), 상태 전이 결과·"권한 없으면 조용히 무시" 동작 100% 동일
- **완료 조건**: `ApprovalServiceCharacterizationTest` 무변경 green

## Phase 2 - 구조 전환 (★★★★☆~★★★☆☆, 4-11 3~9위)

> enum·Converter·DTO·Rich Domain 작업이 서로 얽혀 있어 도메인 단위로 묶어 진행한다. 순서: **enum 도입 → Rich Domain 이관 → DTO 분리**.

### RF-03. 상태/행위/역할 enum + `@Convert` 도입
- **관련 4-11 순위**: 3위(Primitive Obsession, 16점), 4위(상태값·행위값 축 겹침, 12점), 6위(권한 판정 복붙, 10점), 16위(Notice/Schedule 상태값, 4점)
- **선행 조건**: RF-01/RF-02 완료 권장(같은 파일을 또 건드리므로 충돌 최소화), Notice/Schedule 대상은 RF-00 완료 필수
- **작업**:
  - `ApprovalStatus`(0/1/2/3/9), `ApprovalAction`(1/2/3/9), `NoticeStatus`(0/1/9), `ScheduleStatus`(0/1/9), `Role`(1/2/3) enum 도입
  - 전부 `@Convert` + `AttributeConverter`로 기존 정수 코드 매핑
  - `role.canApprove()` 같은 메서드로 `role >= 2` 복붙 3곳 일원화
- **불변 규칙**: **`@Enumerated(STRING)`/`@Enumerated(ORDINAL)` 금지.** DB에는 기존 정수 그대로 저장
- **완료 조건**: 전 도메인 특성화 테스트(Approval 기존 + Notice/Schedule 신규) 무변경 green, DB 저장값 조회 시 정수 동일

### RF-04. Anemic Domain Model → Rich Domain 이관
- **관련 4-11 순위**: 5위 (3 x 4 = 12점)
- **선행 조건**: RF-03 완료 필수 (enum이 있어야 도메인 메서드로 옮길 판정 로직이 명확해짐)
- **작업**: 상태 전이·권한 판정 로직을 `Approval`/`Notice`/`Schedule`/`User` 엔티티(또는 enum) 안으로 이동
- **불변 규칙**: 서비스의 public 메서드 시그니처·리턴값 불변, 상태 전이 결과 동일
- **완료 조건**: 각 도메인 특성화 테스트 무변경 green

### RF-05. 중복 코드 - null 체크 가드 통일
- **관련 4-11 순위**: 7위 (3 x 3 = 9점)
- **선행 조건**: RF-00(Notice/Schedule 특성화 테스트) 완료 필수 - "조용한 무시" 동작을 건드리지 않고 가드만 통일해야 하므로
- **작업**: `findById(...).orElse(null); if(==null) return;` 패턴 3서비스 공통화
- **불변 규칙**: **조회 실패/권한 없음을 예외 없이 조용히 무시하는 동작 자체는 보존.** 가드를 `orElseThrow`로 바꿔 예외를 던지는 것은 동작 변경이므로 금지 - 구조만 통일
- **완료 조건**: 3개 서비스 특성화 테스트 무변경 green (특히 "무시되는 케이스"가 여전히 무시되는지)

### RF-06. DTO 도입 - Entity/Map 직접 노출 제거
- **관련 4-11 순위**: 8위 (3 x 3 = 9점)
- **선행 조건**: RF-01(God Class 분해) 완료 권장 - 책임 분리 후에 DTO 경계를 나누는 편이 명확
- **작업**: 결재 요청(현재 `Map` 수신)/응답(Entity 직접 반환)을 request/response record DTO로 분리
- **불변 규칙**: JSON 구조·필드명은 레거시와 100% 동일 (필드 추가/삭제/이름 변경 금지)
- **완료 조건**: 특성화 테스트에서 API 응답 바디 비교 시 무변경 green

### RF-07. 입력 검증 추가 - `type`/`priority` 범위
- **관련 4-11 순위**: 9위 (3 x 3 = 9점)
- **선행 조건**: RF-03(enum) 완료 필수 - enum 매핑 시점에 유효 범위를 검증하는 구조이므로
- **작업**: 범위 밖 값(`type=9`, `priority=5` 등) 저장 시도에 대한 검증 추가
- **주의**: **이 항목은 "결함이지만 리팩토링 단계에선 손대지 않는" 경계선 사례.** CLAUDE.md 원칙상 "결함을 바로 고치면 동작 변경"이므로, 먼저 특성화 테스트로 "현재는 범위 밖 값도 그대로 저장된다"는 사실을 고정한 뒤, 검증 추가 여부는 **별도 재설계 단계**로 넘기는 것을 권장. 이번 백로그에서는 **보류(Backlog에는 남기되 착수하지 않음)** 로 표시
- **완료 조건**: (착수 시) 검증 추가가 기존 저장 동작을 바꾸지 않는지 특성화 테스트로 사전 확인 후 진행

## 착수 순서 요약

```
RF-00 (특성화 테스트 보강)
  └─ RF-01 → RF-02 (ApprovalService 분해, 세트로 진행)
       └─ RF-03 (enum + Convert, 전 도메인) ─┬─ RF-04 (Rich Domain)
                                              ├─ RF-06 (DTO, RF-14 통합)
                                              ├─ RF-07 (입력 검증, 보류)
                                              └─ RF-17 (Notice.category)
            └─ RF-05 (null 가드 통일, 동작 보존 주의)
            └─ RF-08 (DIP) ─┬─ RF-09 (AuditService)
                             └─ RF-11 (MailMessageFactory)
            └─ RF-10 (금액 임계값, 독립)
            └─ RF-12 (Schedule.create 분리)
            └─ RF-13 (Notice.publish 중첩 완화)
            └─ RF-15 (변수명)
            └─ RF-16 (UserController 계층 분리)
```

- **묶어서 진행**: RF-01+RF-02(같은 세션), RF-08+RF-09+RF-11(`common` 모듈 한 번에), RF-06+RF-14(DTO 도입 한 번에)
- **보류**: RF-07은 CLAUDE.md의 "결함은 리팩토링 단계에서 보존, 재설계 단계로 이관" 원칙에 따라 이번 백로그에서는 사실 고정(특성화 테스트)까지만 하고 실제 검증 로직 추가는 보류
- **검증 게이트**: 각 티켓 완료 시마다 관련 특성화 테스트 전체 실행(`.\mvnw.cmd test`)해서 무변경 green 확인 후 다음 티켓 착수
